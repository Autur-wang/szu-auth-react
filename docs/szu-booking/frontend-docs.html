1.2 技术选型与设计决策

🔧 技术栈选型

纯 HTML5 + CSS3 + Vanilla JavaScript，无任何框架或构建工具依赖。原因：① Demo 阶段快速交付，避免框架学习曲线；② 与深大体育系统集成时无需 SPA 路由冲突处理；③ 便于非工程师团队成员阅读和修改静态 Demo。

生产环境升级方向：Vue 3 + Vite + Pinia（可逐步迁移，已有完整设计文档作为依据）

🎨 设计系统

使用 CSS 自定义属性（Custom Properties）统一管理设计令牌：主色 #1A4189（深大蓝）、警示色 #C1272D（深大红）、背景 #f4f6fb、边框 #e0e5f0。字体：PingFang SC / Microsoft YaHei 中文优先，-apple-system 回退。

📱 响应式策略

移动优先设计。断点：700px 以下切换为单列布局，导航折叠。关键交互（选时段、支付弹窗）在移动端使用底部 Sheet 而非居中 Modal。

⚡ 状态管理

Demo 阶段：所有状态存放于 JS 模块级变量 + localStorage。生产环境：迁移至 Pinia Store，分 authStore / bookingStore / memberStore / notifStore 四个领域 Store。

1.3 关键交互实现说明

① 时段悲观锁 UI（venue-detail.html）

用户点击可用时段 → 前端立即将时段 UI 变为 "锁定中" 状态（防止二次点击），同时调用 POST /slots/{id}/lock 加锁 → 弹出订单确认弹窗并启动 10 分钟倒计时 → 倒计时归零时自动调用 DELETE /slots/{id}/lock 释放锁并关闭弹窗，时段恢复可选。整个倒计时使用 setInterval + 弹窗内显示剩余时间（mm:ss 格式）。

② 短信验证码 60s 倒计时（register.html / forgot-password.html）

点击「获取验证码」→ 调用 POST /sms/send → 成功后按钮禁用，启动 60s setInterval 倒计时显示「重新发送(58s)」→ 归零后恢复可点击。同一组件逻辑复用于注册与找回密码两个页面（提取为独立函数 startCountdown(btnEl)）。

③ 支付结果轮询（payment.html）

展示支付二维码后，前端每隔 3s 轮询 GET /orders/{id}/status → status=paid 时停止轮询并跳转成功页；status=expired（15分钟）时停止轮询并显示超时页；网络错误时最多重试 3 次后提示用户手动刷新。生产环境替换为 WebSocket 推送以降低服务器压力。

④ 学号卡片实时预览（register.html Step 2）

用户输入学号时，通过 oninput 事件实时将学号同步到右侧卡片预览图中（通过 DOM 操作更新 #card-id 文本）。未输入时显示占位符。验证期间卡片显示「验证中」动画（CSS 脉冲效果），验证成功/失败分别切换卡片状态颜色。

⑤ 自动预约队列实时状态（auto-booking.html）

规则列表中每条规则的「状态」字段通过轮询 GET /auto-rules/{id}/status 每 10s 刷新一次（Demo 中使用 setInterval 模拟随机状态切换）。生产环境替换为 WebSocket / SSE，Worker 执行完成后主动推送状态变更。

1.4 页面方案要点
所有页面均采用"先展示骨架 → 异步加载内容"策略，避免白屏。实际联调时在 CSS 中补充 .skeleton 骨架屏样式类，通过 JS 在数据就绪后移除。
⚠️ Demo 页面中所有数据均为静态 Mock，生产环境需逐一替换为真实 API 调用，并补充 Loading / Empty / Error 三种兜底状态。

2.2 核心页面状态流转

🕐 时间段状态机（time_slots.status）

1
available（蓝色）— 默认状态，可点击选择
2
locked（橙色+动画）— 用户点击后10分钟内，显示倒计时，其他用户不可选
3
booked（灰色/条纹）— 已有用户预约完成，不可选
4
closed（深灰）— 管理员关闭或已过期，不可选

locked → available：超时10分钟自动释放 | locked → booked：支付成功

📋 预约记录状态机（bookings.status）

1
pending（黄色）— 已选时段，待支付
2
paid（绿色）— 支付成功，预约确认
3
completed（蓝色）— 场地使用完毕，可写评价
4
cancelled（灰色）— 用户主动取消
5
refunded（紫色）— 退款处理完成

👑 会员状态机

1
free（无会员）— 仅可使用一次自动预约体验
2
trial_used（体验已用）— 触发购买引导 Banner
3
active（会员有效）— 解锁所有自动预约功能
4
expiring（7天内到期）— 展示续费提醒条
5
expired（已到期）— 规则暂停，提示续费

🔔 通知状态

1
unread（蓝点）— 导航栏铃铛显示红点，列表项加粗
2
read（普通）— 点击单条或「全部已读」后变更
3
dismissed（删除）— 点击「×」后从列表移除

未读数量实时同步：通过 WebSocket / 轮询更新导航铃铛角标

4.3 权限控制方案

前端鉴权守卫

每个受保护页面顶部执行 checkAuth()：读取 localStorage 中的 jwt_token → 解析 payload 检查 exp（过期时间）→ 若已过期则清除 token 并跳转 /?redirect=当前URL。

4.5 性能优化方案

🖼️ 图片优化

场地图片使用 WebP 格式（降低 ~30% 体积）；列表页使用 Intersection Observer 实现懒加载（loading="lazy"）；缩略图服务端压缩至 400px 宽，详情页按需加载原图。

📦 资源加载

CSS/JS 文件启用 Gzip/Brotli 压缩；CDN 加速静态资源；关键路径 CSS 内联（首屏样式），非关键 CSS 异步加载；图标使用 CSS emoji 或 SVG，避免图标字体请求。

⚡ 渲染优化

时段网格 16 个格子使用 DocumentFragment 批量插入，避免多次 reflow；滚动监听使用 passive: true；防抖搜索（300ms）避免频繁 API 请求；长列表（预约记录）使用分页而非虚拟滚动（数据量不大）。

🔌 离线体验

生产环境添加 Service Worker：缓存静态资源（HTML/CSS/JS）实现秒开；网络断开时展示离线提示而非白屏；场地列表在 Cache-first 策略下展示旧数据 + 刷新提示。

5.5 回滚预案

触发条件

上线后 30 分钟内：核心转化（支付成功率）下降 >20%；JS 报错率 >5%；P0 功能（登录/预约/支付）不可用；服务器错误率 >1%。

回滚步骤

1
通知 Nginx/CDN 将前端流量切回上一个静态版本目录（v{n-1}/），无需重启服务
2
检查回滚版本是否正常（冒烟测试通过确认），并在群内通知团队
3
保留故障版本文件用于排查，不要删除
4
Sentry 查看具体错误 stack trace，定位根因后修复并重新灰度上线

目标回滚时间：≤ 5 分钟（纯静态前端，无需重新构建）

6.4 前端故障排查手册

🔴 故障：页面白屏 / 无内容

1
打开浏览器开发者工具 Console，查看是否有 JS 报错（通常是 TypeError 或 SyntaxError）
2
Network 标签查看 HTML/JS/CSS 文件是否 200 返回（若 404 说明 CDN 版本切换异常）
3
确认 API 请求是否返回正常（查看 XHR/Fetch 请求，重点看 /auth/me 用户信息接口）
4
若 JS 文件加载失败：检查 CDN 配置，刷新 CDN 缓存；若 API 异常：查看后端 Nginx 日志

🔴 故障：支付页面无响应（二维码展示但无回调）

1
查看轮询接口 GET /orders/{id}/status 是否正常返回（Network 面板）
2
确认支付网关 Webhook 是否配置了正确的 notifyUrl（需后端确认）
3
检查后端 Webhook 接收日志，确认支付网关是否已发送回调
4
临时应对：引导用户等待轮询超时（15分钟），或联系客服手动核对订单状态

🔴 故障：学号绑定大量失败（用户反馈）

1
查看 Sentry student_bind_fail 错误日志，按 reason 分类
2
若 reason=system_error：深大体育系统可能有维护或变更（检查是否可访问），发布系统公告
3
若 reason=wrong_pwd：用户密码问题，排查是否深大统一密码修改导致
4
临时方案：管理后台手动标记用户学号已验证（需人工审核学生证）
